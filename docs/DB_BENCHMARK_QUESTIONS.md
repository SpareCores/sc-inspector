Already identified major technical hurdles requiring a significant amount of work to mitigate:

* Find “good-enough” block storage for each instance type to keep costs low without limiting compute performance with slow disks (potentially up to using RAID0 on IO2 volumes for larger instances to avoid an EBS bottleneck). Take into account the peak network and network storage bandwidth (published by vendors), and other known limiting factors (such as “Maximum supported I/O bandwidth” published by vendors) to save on benchmarking costs, otherwise use available max I/O performance (e.g. IO2 volumes).
* While trying to overprovision disk IO, it’s important to measure actual disk usage so that we can not only report on DB throughput and latency, but also on the max disk IO usage as a proxy what amount of provisioned IOPS capacity can be beneficial for a given instance type.
* Measure and record latency between DBaaS and IaaS products with different placement groups, so we can disclaim the related differences (e.g. when we can place two IaaS in the same rack, but the DBaaS is further away from the IaaS benchmarking client at a given vendor). E.g. extra benchmarks on measuring latency just running “SELECT 1” and documenting the process.
* Benchmarking scenarios, such as OLAP vs OLTP, small to large database sizes, concurrency, read VS write, batch COPY vs INSERT workloads etc – maybe use T-shirt sizes (S, M, L, XL workloads).
